# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

AudioStreamer is a Windows-only WPF desktop app (.NET 10, `net10.0-windows`, single project) that streams system audio between two machines over UDP on a LAN. One instance runs as a **Sender** (captures the local system audio output and transmits it); another runs as a **Receiver** (plays the incoming stream). The role is chosen at runtime via the `Mode` setting.

Originally created in Visual Studio; it builds and debugs entirely from the `dotnet` CLI / VS Code (see `.vscode/`). Requires the .NET 10 Windows Desktop runtime.

## Commands

```powershell
dotnet build                 # Debug build
dotnet build -c Release      # Release build
dotnet run                   # build + launch the WPF app
```

There is no test project. In VS Code, `Ctrl+Shift+B` runs the `build` task and `F5` builds then launches via `.vscode/launch.json` (which points at `bin/Debug/net10.0-windows/AudioStreamer.dll`).

### Installer

```powershell
.\build-installer.ps1                 # publish + compile installer
.\build-installer.ps1 -Version 1.2.3  # override the version for this build
```

`build-installer.ps1` cleans `.\publish\`, runs `dotnet publish -c Release` (framework-dependent), then compiles `installer\AudioStreamer.iss` with **Inno Setup 6** (`ISCC.exe`, found under Program Files) into `installer\Output\AudioStreamer-Setup.exe`. Prereqs: .NET 10 SDK + Inno Setup 6.

Key installer design points:
- **Per-user install** to `%LOCALAPPDATA%\AudioStreamer` (`PrivilegesRequired=lowest`, no UAC). That location is writable, so the app's `config.json` (written next to the exe) works; the shortcut sets `WorkingDir={app}` and the uninstaller deletes `config.json`.
- **Framework-dependent** publish, so the target machine needs the .NET 10 Windows Desktop Runtime. There is intentionally **no installer-side runtime check**: a 32-bit Inno installer reading HKLM is redirected to `WOW6432Node` (where the x64 desktop runtime isn't listed), so the old check false-positived on machines that already had it. The apphost shows its own "install .NET" prompt with a download link on first launch if the runtime is genuinely missing. (Switch to self-contained by adding `--self-contained -r win-x64` in the publish step.)
- **Version** flows from the csproj `<Version>` → published exe → read at compile time by the `.iss` via `GetVersionNumbersString`, so all three stay in sync.
- No "start with Windows" / tray integration (AudioStreamer is a plain windowed app). `AppPublisher`/`MyAppURL` in the `.iss` are **TODO placeholders** — set them before distributing.
- `Resources\icon.ico` is the app/installer icon (`<ApplicationIcon>` in the csproj, `SetupIconFile` in the `.iss`).

## Architecture

**Project layout** — source is grouped into folders by responsibility (the SDK-style csproj globs `**/*.cs`/`**/*.xaml`, so there are no per-file includes to maintain; all types stay in the single flat `AudioStreamer` namespace regardless of folder):
- root: `App.xaml`/`App.xaml.cs` (entry; `App.xaml` keeps `StartupUri="UI/MainWindow.xaml"`), `AssemblyInfo.cs`, project/docs files.
- `UI/` — `MainWindow.xaml`/`.cs`.
- `Core/` — `AudioStreamerLogic.cs` (coordinator + nested `Config`), `IStreamSession.cs`.
- `Streaming/` — `SenderSession.cs`, `ReceiverSession.cs`, `WireProtocol.cs`, and the jitter/loss helpers `ReorderBuffer.cs`, `SequenceLossTracker.cs`, `UnderrunCountingWaveProvider.cs`.
- `Diagnostics/` — `DiagnosticsLog.cs`, `DiagnosticsSnapshot.cs`.
- `Services/` — `StartupService.cs` (HKCU `\Run` toggle).

- **`MainWindow.xaml` / `MainWindow.xaml.cs`** — the only window; a vertical `StackPanel` of labelled fields, each mapping to a `Config` field. Most are `TextBox`es; the three sender **format** fields (Sample Rate, Bits Per Sample, Channels) are selection-only `ComboBox`es (`SampleRateCombo`/`BitDepthCombo`/`ChannelsCombo`) populated from the shared `AudioFormats` catalog via `PopulateFormatDropdowns()` — items are `FormatOption(int Value, string Label)` (label shows e.g. `48000 Hz`, value is the int stored in `Config`). The Channels dropdown's first item is **"Auto (match source)"** (`Config.Channels = 0` / `AudioFormats.AutoChannels`), the default for fresh configs; numeric picks are hard requests. `UpdateConfigFromUI()` / `PopulateUIFromConfig()` sync the UI to `AudioStreamerLogic.CurrentConfig` (`UpdateConfigFromUI` uses `ParseOr` for the `TextBox`es so a blank/invalid field keeps the existing config value instead of throwing; the dropdowns are read via `SelectedValue`, and `PopulateUIFromConfig` preselects by value via `SelectValue`, falling back to the catalog default if a stored value isn't in the list — so selection-only means the format is always valid). The window uses `SizeToContent="WidthAndHeight"` (no fixed `Width`/`Height`) so it auto-fits its content — fields are fixed-width (200) and the window grows to fit, so adding/renaming fields won't clip. `ResizeMode="NoResize"` removes **both** the minimize and maximize caption buttons (Windows only hides the maximize button when the minimize box is also gone — they render as a pair — so a "minimize but no maximize" title bar isn't achievable natively), leaving just the close (X) button. It is also **mode-aware**: `ModeComboBox_SelectionChanged` → `UpdateModePanels()` collapses `SenderPanel` (Host Name, Sender Audio Buffer, Sample Rate, Bits Per Sample, Channels) in Receiver mode and `ReceiverPanel` (the three receiver buffer/latency fields) in Sender mode — Port and Mode are common. Collapsed fields keep their text, so toggling modes is non-destructive. Receiver mode hides the format fields because the receiver derives the format from the packet header, not config.
  - **Tray + startup**: a WinForms `NotifyIcon` (enabled by `<UseWindowsForms>` in the csproj, icon loaded from the embedded `Resources\icon.ico`) provides the tray. Because the title bar has no minimize button (`NoResize`), the **X button hides to the tray** rather than quitting: `Window_Closing` cancels the close and `Hide()`s unless `reallyExit` is set, and only the tray **Exit** item (→ `ExitApp()` sets `reallyExit = true` then `Close()`) actually quits. `StateChanged` → `Hide()` still covers the taskbar-button minimize gesture (the OS minimizes the active window on taskbar click even without a minimize box). Double-click / **Show** restores via `ShowFromTray()`, which also re-asserts `SizeToContent` (toggling through `Manual`) — a window first realised while minimized via `StartMinimized` otherwise returns from the tray oversized with an unpainted region, because `SizeToContent` can't measure a minimized window. `StartButton_Click` delegates to `StartSession(showErrorsAsDialog)`; on `Loaded`, if `Config.StartMinimized` the app docks to the tray and auto-starts the session, routing failures to a tray balloon instead of a dialog. **Start with Windows** is a registry-backed toggle (`StartupService`, HKCU `…\Run`, value `AudioStreamer`) — not in `config.json`; it sits with Start Minimized in `StartupOptionsPanel`, which stays editable while a session runs. The installer's `[Tasks] startup` writes the same Run value so the toggle reflects it on first launch.
  - Because `UseWindowsForms` is on alongside WPF, the WinForms global using makes `Application`/`MessageBox` ambiguous: `App.xaml.cs` qualifies `System.Windows.Application`, and `MainWindow.xaml.cs` aliases `MessageBox` to the WPF type. Fully-qualify WinForms types (`System.Windows.Forms.*`) rather than adding a `using`.
  - **Process priority**: `App.OnStartup` raises the process to `ProcessPriorityClass.AboveNormal` (best-effort, wrapped in try/catch) so the capture / UDP-receive loops are less likely to be preempted under load, reducing audio jitter. AboveNormal is deliberate — `High` risks starving the OS audio service (`audiodg`) and the UI; `RealTime` needs admin.

- **`AudioStreamerLogic.cs`** — a thin **coordinator** (UI-independent). It owns `Config` load/save, the `DiagnosticsLog`, and the `Diagnostics` event, and on `Start()` constructs the right session for the mode. The actual streaming lives in **`SenderSession.cs`** and **`ReceiverSession.cs`** (both implement `IStreamSession` — `{ IsRunning; Start(); Stop(); }`), and the on-wire framing/constants live in **`WireProtocol.cs`**. The coordinator passes each session two delegates — `LogLine` and `Report` — so the session does no IO of its own; `Report` logs the detailed line and raises the `Diagnostics` event.
  - `Config` (nested class) is the full settings model, persisted as `config.json` **next to the executable** — the path is anchored to `AppContext.BaseDirectory` (the install dir, e.g. `bin/Debug/.../config.json`), **not** the current working directory. This matters: "start with Windows" launches from the HKCU `\Run` key with CWD = `C:\Windows\system32` (a Run value can't set a working directory), so a CWD-relative path would try to write `config.json` into `system32` and crash with `UnauthorizedAccessException`. `LoadConfig()` runs in the constructor; on any read/parse failure it writes a fresh default file via `SaveConfig()`. `SaveConfig()` serializes `CurrentConfig` back to disk (`System.Text.Json`, indented; default integer-enum + PascalCase output is byte-compatible with the old Newtonsoft format). Note `LoadConfig()` only ever *adds* missing fields on a parse failure — an existing valid file that lacks a newly-added `Config` field is not rewritten, so the in-memory default is used until the file is next saved (or deleted to regenerate).
  - `Start()` picks `SenderSession` or `ReceiverSession` by `Config.Mode`, calls the session's `Start()`, then publishes it to the `session` field; `IsRunning` reflects `session?.IsRunning`. On a synchronous failure it calls the session's `Stop()` and rethrows so the UI can surface it. `Stop()` snapshots and nulls the field, then delegates teardown to the session.
  - **Sender** (`SenderSession`): `TweakedWasapiLoopbackCapture` (a `WasapiCapture` subclass that sets the `Loopback` stream flag) captures the default render device, then UDP-sends each buffer to `HostName:Port`. The capture build/start lives in `StartCapture()` (factored out of `Start()`, which only opens the UDP socket) so it can be rebuilt without touching the socket. **Auto channel matching**: at every `StartCapture()` call, the default render endpoint's shared-mode mix format is read once via `TryGetDefaultRenderInfo` (before `senderLock` on the first-start path) and its `MixFormat.Channels` is passed to `AudioFormats.ResolveCaptureChannels(config.Channels, nativeChannels)`. "Native" here is the WASAPI shared-mode mix-format channel count — the endpoint's configuration, not the currently-playing content (loopback only ever sees the post-mix stream). Auto (`Config.Channels = 0`) snaps down to the largest catalog channel count ≤ native (native 6→6, 4→2, 2→2), never up-mixing; an explicit count passes through unchanged. The resolved count is used for both `ToWaveFormat` (the capture format) and `ToCode` (the wire format code), so Auto never appears on the wire — the sender always sends a concrete catalog entry. Because `StartCapture()` re-resolves on every rebuild, a device-change recovery automatically follows the new device's channel count. **Auto-recovery**: Windows invalidates the loopback stream when the default render endpoint disappears — most commonly an **HDMI-monitor audio output that goes away while the monitor is in power-saving**, also a default-device change — and NAudio ends its capture thread and raises `RecordingStopped`. `OnRecordingStopped` disposes the dead capture and `RestartCapture()` polls once per second (off the capture thread, first failure logged then suppressed) until a usable output device returns and `StartCapture()` succeeds, re-acquiring the current default device. This replaces the old behaviour where capture silently died until a manual Stop/Start. Teardown goes through `StopCapture()`, which unsubscribes `RecordingStopped` before stopping so an intentional stop isn't mistaken for an unexpected one; `Stop()` clears the session's `isRunning` first so an in-flight rebuild bails, and the `isRunning`-check + `StartCapture()` pair runs under `senderLock` (which `StopCapture()` also takes) so a Stop racing a rebuild never leaves a live capture running. (The `DataAvailable` handler reads the `udpClient` field per callback and no-ops if it's null, so a rebuild/stop can't send on a closed socket.)
  - **Receiver** (`ReceiverSession`): binds a `UdpClient` on `Port`, feeds incoming packets through a `ReorderBuffer` into a NAudio `BufferedWaveProvider` played through `WasapiOut`. It first waits for a packet to read the format header, then builds the buffer/underrun meter/output for that format before streaming. **Per-datagram format tracking**: the receive loop reads the format code on every datagram and compares it to `currentFormatCode`. If the code changes to a valid new entry (e.g. the sender's Auto re-resolved after a device flip), `BuildPipeline` tears down and reconstructs the whole pipeline — `BufferedWaveProvider`, underrun meter, `ReorderBuffer`, and `WasapiOut` — cutting over at the live edge before feeding that datagram, so no old-format-aligned bytes land in the new buffer. The buffer/meter swap is safe because the receive loop is single-threaded (the only thread that touches those fields and calls `AddSamples`); only the `WasapiOut` create/teardown inside `BuildAndPlayOutput` is serialized by `outputLock` (against `Stop()` / `PlaybackStopped` recovery), which now also disposes the outgoing output. An unknown mid-stream code (foreign datagram / version mismatch) is ignored and logged once via `loggedBadFormatMidStream`. Steady-state per-packet cost is a single code-compare with no allocation or lock. **Output auto-recovery** (symmetric to the sender): `WasapiOut` raises `PlaybackStopped` when the receiver's own render device is invalidated (its monitor/HDMI output sleeping, a default-device change); `OnPlaybackStopped` then disposes the dead output off-thread and `RestartOutput()` polls once per second (first failure logged then suppressed) rebuilding via `BuildAndPlayOutput()` until a device returns. `BuildAndPlayOutput()` is the single output-creation point — it `ClearBuffer()`s first so playback resumes at the **live edge** (dropping the second of audio that piled up during the outage) and re-checks `isRunning` **inside `outputLock`** so a Stop racing the rebuild can't resurrect a live `WasapiOut` after teardown (the receiver mirror of the sender's `senderLock` discipline). `ReorderBuffer` restores sequence order before `AddSamples` (NICs deliver the sender's bursty datagrams out of order): in-order packets pass straight through, an early one is held until its predecessor arrives, and a genuinely missing one is skipped once the window fills so playback never stalls. The window depth is `ReceiverSession.ComputeReorderWindow(format)` — scaled by the format's byte rate (`AverageBytesPerSecond`, so it covers sample rate × bit depth × channels) to keep the give-up *time* ~constant (~40 ms) across formats: 8 packets at 16-bit/48 kHz/stereo, 16 at 32-bit, capped at 64. A fixed count would be too shallow for hi-res/multichannel.
  - **Latency cap (drift control)**: the receiver's audio backlog (`BufferedWaveProvider.BufferedDuration`) *is* the current audio-behind-video delay. Because the sender and receiver run on independent sound-card clocks, that backlog drifts upward over time. After each `AddSamples`, if the backlog exceeds `Config.ReceiverMaxLatencyMilliseconds` (default 150; set ≤0 to disable) the receive loop trims only the excess down to a frame-aligned low-water mark (half the cap) by read-and-discarding from the buffer — keeping playback continuous rather than emptying it with a `ClearBuffer()` click. `ReceiverAudioBufferMillisecondsLength` (default 1000) is only the overflow ceiling/jitter headroom and must stay comfortably above the cap. The receive loop and the sender's `DataAvailable` handler build a `DiagnosticsSnapshot` once per second (backlog, min backlog, pkts/s, KB/s, lost/s, reorder/s, underrun/s, overflow/s, resync/s) and call `Report()`, which logs the detailed line and raises the `Diagnostics` event. `lost/s` and `reorder/s` come from `SequenceLossTracker`, which unwraps the per-datagram sequence byte and classifies a gap as a **late arrival** (`reorder/s` — the datagram shows up out of order within a 32-packet window) vs a **true loss** (`lost/s` — never arrives); a clean wire with `reorder/s` ticking points at the NIC/host (e.g. 2.5 GbE receive-side scaling) reordering bursty sends, not packet loss. The drift cap only fires in one direction (backlog *growing* → `resync/s`); the opposite drift — receiver clock faster, backlog *draining* to empty — shows up as **`underrun/s`**, counted by `UnderrunCountingWaveProvider`, a transparent pass-through wrapping the `BufferedWaveProvider` that `WasapiOut` plays from. It increments when a render-thread `Read` can't be fully satisfied (the buffer zero-fills the gap → a brief silent "drop" the receive loop can't otherwise see). `min` backlog (lowest backlog sampled in the interval, before each `AddSamples`) trends toward 0 as the lead-in to an underrun. A steadily climbing `backlog` is the signature of growing-direction clock drift; a `min` near 0 with `underrun/s` ticking is the shrinking-direction signature. Diagnostics are decoupled from streaming: `DiagnosticsLog` (constructed with `Path.Combine(AppContext.BaseDirectory, "diagnostics.log")`, so it sits next to the exe / `config.json` regardless of working directory — see the `config.json` note above) timestamps and **enqueues** each line; a single background thread does all file IO + console mirroring and rotates at 1 MB → `diagnostics.1.log`. The audio threads never touch disk. `MainWindow` subscribes to `Diagnostics` and shows `snapshot.ToCompactLine()` in the dim `DiagnosticsText` readout under the status line (via `Dispatcher.BeginInvoke`), visible only while running.

- **Wire protocol** (`WireProtocol.cs`, a static class shared by both sessions): every UDP packet is a **2-byte header + raw PCM audio**. Byte 0 (`FormatHeaderBytes` = 1) is a **format code** — an index into `AudioFormats.Formats`, the shared 72-entry catalog (cartesian product of the allowed sample rates × bit depths × channels) — written by `WriteFormatCode`/read by `ReadFormatCode`. The sender derives the code from `Config` once at capture start (`AudioFormats.ToCode`); the receiver maps it back to a `WaveFormat` (`AudioFormats.FromCode`) and **rejects an out-of-range code** (logs once, keeps waiting — the item-16 guard). Encoding the format as a catalog index rather than packed fields makes invalid/unrepresentable combinations impossible on the wire and fixes the old `sampleRate/1000` byte-truncation (44.1 kHz used to arrive as 44 kHz). Byte 1 (`SequenceByteOffset`) is a wrapping per-datagram sequence number the receiver uses to count lost/reordered packets (`lost/s` in diagnostics). `HeaderBytes` (2) is the audio offset on both sides. The sender slices each capture into whole-frame datagrams of ≤ `MaxUdpAudioBytes` (1440) audio bytes so each datagram stays under the Ethernet MTU (no IP fragmentation) and stays frame-aligned (a lost datagram drops whole frames). `WireProtocol` also holds the socket tunables (`SocketBufferBytes`, `SIO_UDP_CONNRESET`). The catalog order is part of the wire contract, so **both sides must run the same version** (already required).

## Start/Stop lifecycle

`Start()` is non-blocking: the sender's capture runs on NAudio's own `DataAvailable` thread and the receiver runs via `Task.Run`, so both return immediately to the UI thread. Each session owns its own nullable resources — the sender holds `udpClient`/`waveSource` (guarded by `senderLock`); the receiver holds `udpClient`/`socket`/`wasapiOut`/`cts` (output guarded by `outputLock`) — assigned on `Start()` and torn down in the session's `Stop()` (clear `isRunning` → cancel token → stop/dispose capture or output → close socket → null everything so a fresh Start works). Closing the socket in the receiver's `Stop()` is what unblocks its blocking `ReceiveFrom`; the cancellation-token check then exits the loop.

`Start()` guards on `IsRunning` (a no-op if already running, preventing a double-start that would crash the receiver on a port-in-use bind), and wraps the launch in try/catch: on failure it calls `Stop()` to clean up the partial session and rethrows. Synchronous failure points (receiver port-in-use, sender `IPAddress.Parse` on a bad Host Name, `StartRecording` with no audio device) throw before any background task starts, so they surface to the caller. `StartButton_Click` catches that and shows a `MessageBox` rather than crashing.

UI state is driven by `SetRunningState(bool)`: Start disabled / Stop enabled / `SettingsPanel` locked / status line set while running, and the inverse when idle. Persistence happens on Start (`UpdateConfigFromUI` → `SaveConfig`) and on window close (`Window_Closing` → `UpdateConfigFromUI` → `SaveConfig` → `Stop`). A *background-thread* session failure is not surfaced to the UI — it keeps showing "Running". Both sides now self-heal from a lost render device (sender capture via `RestartCapture`, receiver output via `RestartOutput`), so "Running" stays truthful across a monitor-sleep/device-change once the device returns; only a genuinely unrecoverable mid-stream failure (e.g. the receiver's bind socket dying) would still leave a stale "Running".

Do **not** reintroduce `Console.ReadKey` keep-alive loops here — an earlier version of this code used them, but with no console attached they throw `InvalidOperationException` and would block the UI thread. The remaining `Console.WriteLine` calls are harmless diagnostics (visible when run via `dotnet run`, which attaches a console).

- Nullable reference types are enabled. The session fields are guarded with `?.` throughout; this is intentional, not a bug to "fix" by force-initializing them.
